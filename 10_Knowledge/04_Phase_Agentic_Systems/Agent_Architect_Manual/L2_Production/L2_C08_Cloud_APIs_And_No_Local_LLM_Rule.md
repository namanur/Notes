# Chapter [08]: Cloud APIs And The No Local LLM Rule
> Layer: L2 Production | Prerequisite: L1_C07_TDD_And_Execution_Loops

## [1] Concept Explanation
In the context of this **Harness**, we enforce a strict **No Local LLM Rule**. This is not a preference, but an architectural necessity driven by hardware constraints and the goal of reliable **Action**. Local LLM inference—running models like Llama 3 or DeepSeek on your own machine—requires significant VRAM (GPU memory) and compute power. On a Dell Ubuntu Server with 4GB RAM already hosting ERPNext, local inference would lead to system instability, high latency, and frequent OOM (Out of Memory) crashes.

By offloading **Reasoning** to **Cloud APIs** (Gemini, OpenRouter), we ensure that the local system resources are reserved for what the Dell server does best: maintaining **Memory** (SQLite), executing **Tools** (Python/Bash), and managing the **MCP** boundary. The cloud provides the brain; the local server provides the body and the persistent state.

## [2] Why This Layer Exists
Layer 2 (Production) is about reliability. In a lab setting (L0-L1), a slow or failing model is an annoyance. In **Production**, where we aim for **Automated B2B digital services**, a model that hangs because the local GPU is throttled is a business failure. We use Cloud APIs to guarantee a "Quality of Service" (QoS) that local hardware at this scale cannot provide. This layer formalizes the **FALLBACK_CHAIN.sh** logic to handle the inevitable reality of cloud life: rate limits.

## [3] Stack Integration
Our stack relies on the **Gemini CLI** as the primary engine for **Perception** and **Reasoning**. However, since we are on the free tier, we are bound by strict **Context Window** quotas and RPM (Requests Per Minute) limits.
- **Primary:** Gemini 1.5 Flash (via Gemini CLI). High speed, generous context, free.
- **Secondary:** OpenRouter (via `FALLBACK_CHAIN.sh`). Used when Gemini returns a `429 Too Many Requests` error.
- **State Tracking:** Every **Action** and **Output** is logged to the SQLite database to monitor token usage and cost, even when falling back to paid models on OpenRouter.

## [5] Hands-On Execution
In this exercise, you will intentionally trigger a rate limit and verify that your **Harness** correctly transitions to a fallback model.

### Task: The Fallback Stress Test
1. **Identify the Script:** Locate your `_SYSTEM/FALLBACK_CHAIN.sh`.
2. **Artificial Throttle:** To simulate a rate limit without actually burning your quota, you can temporarily edit the script to force the `if` condition to true, or run a loop that hits the Gemini API rapidly with complex prompts.
3. **Execution:** Run the following command:
   ```bash
   bash _SYSTEM/FALLBACK_CHAIN.sh "Analyze this accounting ledger for discrepancies: [Insert 100 lines of dummy data]"
   ```
4. **Observation:** Watch the terminal **Output**. You should see the message: `[FALLBACK] Gemini limited → trying DeepSeek R1`.
5. **Validation:** Verify that the final response is generated by DeepSeek R1 (OpenRouter) instead of Gemini.

**Proof of Completion Path:**
Run `bash _SYSTEM/FALLBACK_CHAIN.sh "Test fallback"`. Capture the terminal output where the `[FALLBACK]` message appears and save it as `L2_C08_fallback_log.txt` in your current daily log directory.

## [7] Upgrade Trigger
**The Silent 2 AM Failure:** You will know you need to move beyond the free tier when your automated workflows begin to fail silently at 2 AM because of rate limits. If your **SQLite Memory** shows a pattern of "Gemini limited" logs exceeding 15% of total calls, it is time to attach a billing account to Google Cloud or increase your OpenRouter credits to ensure 100% **Action** availability.
